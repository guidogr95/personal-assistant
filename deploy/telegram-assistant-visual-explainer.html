Telegram Assistant

A self-hosted, containerized AI personal assistant that lives in Telegram. It remembers conversations, manages tasks, transcribes videos, and proactively checks in — all running on a single 2GB VPS.

v0.1.0 Python 3.12 MVP + Active Dev Docker Compose 2GB RAM Single-User

Architecture

Seven Docker containers on a single bridge network. The bot is the orchestrator — everything else is a sidecar. No external load balancer, no Kubernetes, no cloud dependencies beyond the LLM API.

Data & Component Flow

A message from the user flows through authorization, session management, LLM inference, tool execution, and back to Telegram. The video transcription feature adds a background worker that processes jobs asynchronously.

Module Breakdown

The codebase follows DDD layering: domain (entities/VOs), application (use cases), infrastructure (adapters), and presentation (Telegram handlers). Each module is self-contained with its own exceptions and logging context.

Presentation

Telegram Layer

aiogram long-polling dispatcher with router-based handlers. AllowedUserMiddleware gates all updates to a single user. Markdown → Telegram HTML conversion with plain-text fallback.

telegram/handlers/message.py, telegram/bot.py, telegram/formatting.py
⚠️ Markdown parse errors silently fall back to plain text — formatting may be lost.
Agent

LLM + Tools

pydantic-ai Agent factory with OpenAI-compatible backend. System prompt is a single source of truth (DB on first run, 80-line inline fallback). 8 toolsets auto-discovered via @tool decorator. Dependencies injected through RunContext[AgentDeps] — zero module-level globals. MCP memory integration via HTTP.

agent/domain/agent.py, agent/domain/deps.py, agent/tools/registry.py, agent/tools/
⚠️ Per-user asyncio.Lock serializes turns — concurrent messages wait, they don't race.
Domain

Conversation

Session aggregate (ACTIVE/CLOSED) with Turn value objects. SQLite repositories with aiosqlite. Session stores serialized pydantic-ai message history as a JSON blob.

conversation/domain/session.py, conversation/domain/turn.py, conversation/infrastructure/sqlite_repositories.py
⚠️ message_history_json is a serialized blob — schema changes require migration logic.
Feature

Video Transcription

YouTube (captions → audio → ASR), TikTok/Instagram (audio → ASR). FIFO queue with single worker. Groq Whisper primary, local Whisper tiny fallback. bgutil sidecar for YouTube bot detection bypass.

video/application/extract_video_transcript.py, video/application/transcription_queue.py, video/infrastructure/
⚠️ Queue is in-memory only — jobs lost on restart. Groq has 25MB file limit.
Feature

Scheduler

APScheduler with cron (recurring) and fire_at (one-off) checkins. Auto-disable on max_runs. Timezone-aware. All exceptions caught to prevent scheduler crash.

scheduler/domain/scheduled_checkin.py, scheduler/application/run_checkin.py, scheduler/infrastructure/apscheduler_registry.py
⚠️ Exceptions are swallowed — errors only visible in logs, not to user.
Feature

Notes

Markdown vault on filesystem (/srv/notes). CRUD operations with case-insensitive search. Syncthing syncs to other devices. File mtime used for timestamps.

notes/domain/note.py, notes/infrastructure/markdown_repository.py
⚠️ Filename collisions overwrite existing files — no uniqueness enforcement.
Feature

Research

Web search via SearXNG (local). Page fetch via Jina. Browser automation via Rebrowser Playwright. Results fed back to LLM for synthesis.

research/application/search_web.py, research/infrastructure/

Memory (MCP)

MCPToolset over Streamable HTTP to mcp-memory-service. LLM calls memory_store and memory_retrieve directly. SQLite persistence with daily backups.

memory/infrastructure/mcp_client.py

Shared

Pydantic Settings from .env. structlog with key-value structured output. UTC-aware datetimes. Custom exception hierarchy per domain.

Key Concepts & Mental Model

These are the things you must hold in your head to work on this codebase effectively. Each invariant is enforced somewhere; violating assumptions here leads to subtle bugs.

  1. One Active Session Per User The bot is single-user, but the session model supports multi-user. Auto-created on first message. Closed sessions are read-only. If no active session exists, NoActiveSessionError is raised and caught to auto-create.
  2. Message History Is a Serialized Blob Session.message_history_json stores the full pydantic-ai message list as JSON. It's deserialized on every turn, mutated, and re-serialized. This is the LLM's context window — schema changes break history.
  3. Per-User Lock Prevents Race Conditions asyncio.Lock per user_id in run_turn.py:82. Two rapid messages serialize — the second waits for the first to complete. Without this, turns interleave and history is corrupted.
  4. Container-First Development Everything runs in Docker. The host is irrelevant. /app/.venv/bin/ must be in PATH. Services talk via container names (e.g., http://bgutil:4416). Never use 127.0.0.1 for cross-container communication.
  5. Tool Sentinel Interception run_turn checks for DELETE_CONFIRM_SENTINEL in the LLM response. If found, it sends a confirmation prompt instead of the raw text. This is how the delete-note confirmation flow works — silently, without the LLM knowing.
  6. Queue Jobs Are Ephemeral The transcription queue is an asyncio.Queue in memory. On bot restart, pending jobs vanish. Users won't know why their transcript never arrived. Acceptable for a personal bot, but document this limitation.
  7. YouTube Bot Detection Requires Three Things To download from YouTube on a datacenter IP: (1) bgutil sidecar for PO tokens, (2) mweb player client, (3) Node.js ≥22 for EJS challenge solving. Missing any one = "Sign in to confirm you're not a bot."
  8. Groq Falls Back to Local Whisper Silently If the Groq API key is missing, quota is exceeded, or the file is >25MB, the system falls back to local Whisper tiny (39MB model, ~400MB RAM). The user is not notified which path was taken — only the service_used field in metadata reveals it.

Extensibility

The codebase is designed so that adding new capabilities requires touching exactly one file. No edits to main.py, no manual wiring, no registration lists. The factory pattern and auto-discovery handle everything.

How Dependency Injection Works

All agent tools receive their runtime dependencies through RunContext[AgentDeps] instead of module-level globals. AgentDeps is a dataclass created in main.py and passed to every tool invocation automatically by pydantic-ai. 

# src/assistant/agent/domain/deps.py
from dataclasses import dataclass
from aiogram import Bot
from apscheduler.schedulers.asyncio import AsyncIOScheduler

@dataclass
class AgentDeps:
    scheduler: AsyncIOScheduler
    checkin_repo: ScheduledCheckInRepository
    prompt_repo: PromptRepository
    note_repo: NoteRepository
    bot: Bot
    vikunja_client: VikunjaClient
    searxng_client: SearXNGClient
    jina_client: JinaClient
    rebrowser_client: RebrowserClient
Rule: If your new tool needs a new dependency, add it to AgentDeps and instantiate it in main.py. All existing tools receive the updated container automatically.

Adding a New Agent Tool

Create one file in src/assistant/agent/tools/:

# src/assistant/agent/tools/weather_tools.py
from __future__ import annotations

import structlog
from pydantic_ai import RunContext

from assistant.agent.domain.deps import AgentDeps
from assistant.agent.tools.registry import tool

logger = structlog.get_logger()


@tool(category="🌤️ Weather")
async def get_weather(ctx: RunContext[AgentDeps], city: str) -> str:
    """Get the current weather for a city.

    Args:
        city: City name, e.g. "London" or "New York".
    """
    logger.info("weather_tool_called", city=city)
    return f"The weather in {city} is sunny and 22°C."

That's it. The @tool(category="...") decorator registers the function in ALL_TOOLS and declares its display category. On startup, main.py iterates over ALL_TOOLS and calls agent.tool(fn) for each one. The /tools command discovers categories dynamically — no central map to maintain. No other file needs to change.

What happens under the hood:
  1. Module is imported when main.py runs (or earlier if something else imports it)
  2. @tool(category="...") appends the function to ALL_TOOLS and records its category
  3. main.py calls create_agent(), then loops for tool_fn in ALL_TOOLS: agent.tool(tool_fn)
  4. The /tools command reads categories from the registry and groups tools automatically
  5. The tool is callable by the LLM immediately

Adding a New Telegram Command

Create one file in src/assistant/telegram/handlers/:

# src/assistant/telegram/handlers/weather_commands.py
from __future__ import annotations

import structlog
from aiogram import Router
from aiogram.filters import Command
from aiogram.types import BotCommand, Message

from assistant.telegram.formatting import send_message

logger = structlog.get_logger()

router = Router()

COMMANDS = [
    BotCommand(command="weather", description="Get current weather"),
]


@router.message(Command("weather"))
async def cmd_weather(message: Message) -> None:
    """Handle /weather command."""
    if not message.text:
        return
    await send_message(message, "It's sunny! ☀️")

That's it. discover_routers() finds the router automatically. discover_commands() collects COMMANDS for the Telegram menu. No other file needs to change.

Router registration order: The catch-all message.py router must be last. discover_routers() returns (module_name, router) tuples and main.py sorts them by a priority map so errors is first and message is last. New command routers default to priority 50 (middle) and are registered before the catch-all.

Adding a New Domain Module

For a full feature (domain + application + infrastructure), follow the existing DDD structure:

src/assistant/weather/
  __init__.py
  domain/
    __init__.py
    weather_report.py          # Entity / Value Object
    weather_repository.py      # Protocol
  application/
    __init__.py
    fetch_weather.py           # Use case
  infrastructure/
    __init__.py
    openweather_client.py      # Adapter

Then wire the new repository/client into AgentDeps and pass it to any tool that needs it. The domain layer never imports infrastructure — only Protocols and value objects.

Architecture Decisions

These patterns were chosen after evaluating alternatives. Full ADRs are in local_code/docs/extensibility-refactor/architecture/.

Decision Chosen Rejected
Tool DI RunContext[AgentDeps] Module-level globals, class-based toolsets, functools.partial
Scheduler→agent cycle Callback pattern (run_agent) Event bus, mediator class, mid-file import
Agent creation Factory in main.py Module-level agent instance
Tool registration @tool + ALL_TOOLS registry Manual registration list, plugin system
Router discovery pkgutil.iter_modules + priority sort Manual router list in main.py
Command menu COMMANDS export per handler Monkey-patch aiogram Router, manual command list

Getting Started

1. Clone & Configure

git clone <repo-url> telegram-assistant
cd telegram-assistant
cp .env.example .env
# Edit .env with your API keys and tokens

2. Deploy

# From project root — .env must be here
make deploy

# Or explicitly:
docker compose --env-file .env -f deploy/docker-compose.yml up -d --build

3. Verify

# Check all services are healthy
docker compose --env-file .env -f deploy/docker-compose.yml ps

# View bot logs
make botlogs

# Run tests
make test

4. Test Video Transcription (Inside Container)

# Copy test script into container
docker cp development/playground/test_full_transcription.py \
  deploy-bot-1:/tmp/

# Run it — tests the full pipeline
docker exec -it deploy-bot-1 /app/.venv/bin/python3 \
  /tmp/test_full_transcription.py \
  "https://www.youtube.com/watch?v=dQw4w9WgXcQ"
Production checklist. Before declaring the bot ready:
  • .env is never committed (check .gitignore)
  • All services show Up or healthy in docker ps
  • Video transcription test passes from inside the container
  • Bot responds to Telegram messages without TelegramConflictError
  • Only one bot instance is running (check docker ps)
